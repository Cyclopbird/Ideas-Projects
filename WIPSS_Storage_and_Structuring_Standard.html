Storage and Structuring Standard for Ideas & Projects

This document defines the recommended system for organizing, storing, and interlinking scientific ideas and projects to ensure easy discoverability, clarity, scalability, and interoperability â€” optimized for both human and AI consumption.

1. Unique IDs

Every idea/project must have a unique, persistent identifier that serves as a stable reference point. Suggested format:

Idea_[Topic]_[IncrementalNumber]

Example: Idea_Fusion_001

Use consistent naming to enable automated indexing and linking.

2. File Naming & Location

  • Save each idea as an individual .html file named exactly after its unique ID, e.g. Idea_Fusion_001.html.
  • Organize all idea files in a dedicated directory, e.g. /Ideas-Projects/.
  • Maintain a central index.html that lists all ideas with clickable links, unique IDs, creation dates, and concise summaries.

3. Content Structure

  • Title: Clear, descriptive, and concise.
  • Author(s): Full name(s) and affiliation(s).
  • Description: Brief summary highlighting the core concept.
  • Detailed Content: Structured logically with semantic HTML headings and paragraphs.
  • Tags/Keywords: Relevant keywords or topics for categorization and search optimization.
  • Related Ideas: Explicitly link to related idea files using their unique IDs.

4. Metadata

Embed machine-readable metadata using JSON-LD with schema.org vocabularies, preferably CreativeWork or derived types. This supports automated indexing by search engines and AI systems.

Recommended metadata fields include:

  • identifier, name, author, dateCreated
  • keywords, about, relatedLink (to link related ideas/projects)
  • version, status (e.g. draft, published)
  • Optionally, citation, license, and custom data points relevant for AI (e.g. impactScore)

5. Versioning and Updates

  • Maintain explicit version information within each idea's metadata (e.g. semantic versioning 1.0.0).
  • Record update timestamps.
  • Keep a changelog embedded or as a separate linked file.
  • Reflect updates in the central index to keep the repository current.

6. Cross-Linking and Clustering

Enable semantic connections by linking related ideas/projects via their unique IDs using:

  • Explicit hyperlinks in the content.
  • Machine-readable relatedLink arrays in JSON-LD.
  • Tags and categories for thematic grouping.

This enables automated clustering, graph visualizations, and discovery.

7. Hosting and Accessibility

Host the collection on platforms that support static site hosting (e.g. GitHub Pages) or decentralized storage (e.g. IPFS) to ensure global accessibility and permanence.

8. International and Open Standards

Adhere strictly to international open data standards (e.g. JSON-LD, schema.org) and semantic web best practices to maximize interoperability and future-proof your system.

9. Collaboration and Contribution

Define clear contribution guidelines for community input, including:

  • Standardized submission formats
  • Review and validation processes
  • Version control and conflict resolution

10. Standard Identification

All ideas/projects stored following this system must declare compliance with the Wieser Idea-Project Storage Standard (WIPSS) by including metadata with the standard name and version. This enables automated recognition, validation, and interoperability.

Example JSON-LD metadata snippet:

{
  "@context": "https://schema.org",
  "@type": "CreativeWork",
  "identifier": "Idea_Example_001",
  "name": "Example Idea",
  "author": {
    "@type": "Person",
    "name": "Martin Wieser"
  },
  "dateCreated": "2025-05-30",
  "keywords": ["example", "idea", "project"],
  "storageStandard": {
    "@type": "DefinedTerm",
    "name": "Wieser Idea-Project Storage Standard",
    "version": "1.0"
  }
}

Visible attribution is recommended, e.g.:
This idea/project page is created following the Wieser Idea-Project Storage Standard (WIPSS) version 1.0.
